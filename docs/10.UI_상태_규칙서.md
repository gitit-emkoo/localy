# UI 상태 규칙서

## 문서 개요
이 문서는 같은 상태라면 앱 전체에서 같은 방식으로 보이게 만드는 규칙서다.

예를 들면 이런 문제를 방지한다.

- 버튼이 언제 활성인지 애매함
- 로딩 중인데 여러 번 클릭됨
- 빈 상태와 에러 상태가 섞임
- 모달 / 토스트 타이밍이 화면마다 다름
- 같은 데이터인데 화면마다 표시 규칙이 다름

즉 이 문서는 **같은 상태라면 앱 전체에서 같은 방식으로 보이게 만드는 규칙서**다.

---

## 10-1. 문서 목적
이 문서는 앱의 모든 화면에서 공통적으로 적용되어야 하는 UI 상태 표현 규칙을 정의한다.

정의 대상은 아래와 같다.

- 버튼 활성 / 비활성 / 로딩 규칙
- 로딩 상태
- 빈 상태
- 에러 상태
- 성공 피드백
- 모달 / 바텀시트 / 토스트 노출 타이밍
- 리스트 / 카드 / 입력폼의 상태 표시 방식

이 문서는 **디자인 일관성과 구현 일관성**을 동시에 위한 문서다.

---

## 10-2. 기본 원칙

### 원칙 1. 상태는 화면마다 새로 정의하지 않는다
같은 상황은 모든 화면에서 같은 방식으로 표시한다.

예:
- 저장 성공 → 항상 토스트
- 작성 중 이탈 → 항상 모달
- 재시도 가능한 실패 → 항상 에러 상태 + 다시 시도 버튼

### 원칙 2. 버튼은 3상태만 쓴다
모든 주요 버튼은 아래 3상태만 사용한다.

- `active`
- `disabled`
- `loading`

별도의 애매한 상태를 만들지 않는다.

### 원칙 3. empty state와 error state를 구분한다
둘은 전혀 다른 상태다.

- `empty` = 정상인데 데이터가 없는 상태
- `error` = 비정상적으로 불러오지 못한 상태

둘을 같은 디자인으로 처리하지 않는다.

### 원칙 4. 피드백은 짧고 즉각적이어야 한다
성공 / 실패 피드백은 길게 설명하지 않는다.

좋은 예:
- 제출이 완료되었어요
- 저장에 실패했어요
- 다시 시도해 주세요

### 원칙 5. 글로벌 기준으로 텍스트보다 구조와 상태 색이 더 명확해야 한다
즉 아래 같은 시각적 신호를 우선한다.

- 버튼 변화
- 칩 색상
- 진행 상태 바
- disabled 처리
- 로딩 스피너

## 10-3. 버튼 상태 규칙

### 버튼 공통 상태 정의

#### 1) Active
- 클릭 가능
- 브랜드 기본 색상 사용
- 그림자 / 강조 가능

#### 2) Disabled
- 클릭 불가
- 흐린 배경 / 텍스트
- 터치 이벤트 없음

#### 3) Loading
- 클릭 불가
- 스피너 표시
- 중복 클릭 금지
- 버튼 크기와 위치는 유지

### 버튼 상태별 공통 규칙

#### active
- 터치 가능
- 다음 액션 명확
- hover / pressed 상태 정의 가능

#### disabled
- 이유가 화면 내에서 충분히 추론 가능해야 함
- 필요 시 설명 문구 동반

예:
- 결과 보기 버튼 `disabled`
- 아래 문구:
  - 상대 팀원의 업로드가 완료되면 결과 보기가 활성화됩니다

#### loading
- 텍스트 대신 스피너 또는 스피너 + 텍스트

예:
- 제출 중...
- 저장 중...

- 로딩 중 다른 버튼 연타 방지

---

## 10-4. 화면별 주요 버튼 상태 규칙

### A. 홈 > 오늘의 미션 카드 버튼

#### 상태 1. 매칭 전
- 버튼명: `매칭 시작`
- 상태: `active`

#### 상태 2. 매칭 중
- 버튼명: `매칭 중...`
- 상태: `disabled` 또는 `loading` 성격
- 실제 클릭 불가

#### 상태 3. 매칭 완료
- 버튼명: `오늘의 미션 참여`
- 상태: `active`

#### 상태 4. 매칭 실패
- 버튼명: `다시 매칭하기`
- 상태: `active`

### B. 오늘의 팀 미션 화면 메인 버튼

#### 상태 1. `not_started`
- 버튼명: `미션하기`
- 상태: `active`

#### 상태 2. `self_submitted_waiting`
- 버튼명: `결과 보기`
- 상태: `disabled`

#### 상태 3. `peer_submitted_waiting`
- 버튼명: `미션하기`
- 상태: `active`

#### 상태 4. `ready_to_view`
- 버튼명: `결과 보기`
- 상태: `active`

### C. 미션 참여 화면 제출 버튼

- `draft_empty`
  - `제출하기`
  - `disabled`

- `draft_partial`
  - `제출하기`
  - `disabled`

- `ready_to_submit`
  - `제출하기`
  - `active`

- `submitting`
  - `제출 중...`
  - `loading`

### D. 결과 보기 화면 액션 버튼

- 표현 선택
  - `active`
  - 클릭 시 바텀시트

- 저장
  - `active`
  - 토글형

- 공유하기
  - `active`
  - OS 공유 or 복사

- 아이콘 반응
  - `active`
  - 단, 요청 중에는 순간적으로 `disabled` 가능
  - MVP: **같은 아이콘 재탭 = 취소**, **다른 아이콘 탭 = 변경**(선택 하이라이트는 현재 유효 반응 1개)

### E. 프로필 편집 / 가입 프로필 설정 버튼

- 필수값 부족
  - `disabled`

- 필수값 충족
  - `active`

- 저장 중
  - `loading`

## 10-5. 로딩 상태 규칙

### 로딩 상태 종류
앱에서 로딩은 3종류로 나눈다.

#### 1. 전체 화면 로딩
대상:
- 앱 첫 실행
- 세션 확인
- 첫 데이터 진입

표현:
- 화면 중앙 로더 또는 스켈레톤
- 배경은 유지

#### 2. 섹션 로딩
대상:
- 홈의 Board 미리보기
- World 카드 목록
- My 활동 요약

표현:
- 카드 / 리스트 스켈레톤
- 나머지 화면은 유지

#### 3. 액션 로딩
대상:
- 제출 버튼
- 저장 버튼
- 매칭 시작
- 설정 저장

표현:
- 해당 버튼만 `loading`
- 다른 영역은 유지

### 스켈레톤 사용 규칙
스켈레톤은 아래에 사용 가능하다.

- Board 카드 목록
- 결과 카드 상세의 이미지 / 텍스트 자리
- My 활동 요약
- World 리스트

스켈레톤은 실제 레이아웃 구조를 그대로 따라야 한다.  
즉, 로딩 중에도 레이아웃 점프가 적어야 한다.

## 10-6. Empty state 규칙

### 공통 원칙
empty state는 실패가 아니다.  
“데이터가 아직 없음”을 자연스럽게 보여줘야 한다.

각 empty state는 반드시 아래를 포함한다.

- 제목 / 한 줄 설명
- 다음 행동 버튼 1개

### 대상 화면별 empty state

#### 1. 내 결과 카드 목록
문구:
- 아직 참여한 결과 카드가 없어요

버튼:
- 오늘의 미션 참여하러 가기

#### 2. 저장한 카드 목록
문구:
- 아직 저장한 카드가 없어요

버튼:
- Board 둘러보기

#### 3. Board
문구:
- 아직 공개된 결과 카드가 없어요

버튼:
- 홈으로 가기

#### 4. World
문구:
- 아직 참여 현황이 쌓이지 않았어요

버튼:
- Board 보기 또는 홈으로 가기

#### 5. 알림
문구:
- 아직 도착한 알림이 없어요

버튼:
- 홈으로 가기

### empty state 디자인 원칙
- 과한 일러스트보다 간단한 아이콘 / 텍스트
- 한 화면을 다 먹기보다 카드형 empty도 가능
- 다음 행동이 분명해야 함

---

## 10-7. Error state 규칙

### 공통 원칙
error state는 아래 요소를 포함해야 한다.

- 실패 문구
- 재시도 버튼
- 필요 시 이전 화면으로 가는 보조 선택지

### error state 대상
- 결과 카드 로드 실패
- World 데이터 로드 실패
- 알림 로드 실패
- 프로필 로드 실패
- Board 로드 실패

### 공통 문구 패턴
- 불러오지 못했어요
- 다시 시도해 주세요

예:
- 결과를 불러오지 못했어요
- 다시 시도해 주세요

### 에러 UI 규칙
- empty state와 다른 색 / 아이콘 사용 가능
- “데이터 없음”과 혼동되지 않게 해야 함

---

## 10-8. 토스트 규칙

### 토스트 사용 목적
토스트는 사용자를 멈추지 않고 짧게 알려야 하는 상황에만 쓴다.

### 토스트 사용 대상
- 제출 완료
- 카드 저장 완료
- 저장 해제 완료
- 표현 선택 완료
- 공유 / 복사 완료
- 반응 완료(선택)
- 프로필 저장 완료

### 토스트 문구 원칙
- 1문장
- 15자 내외 우선
- 현재형 또는 완료형

예:
- 제출이 완료되었어요
- 카드가 저장되었어요
- 표현을 남겼어요
- 링크가 복사되었어요

### 토스트 노출 규칙
- 하단 또는 상단 고정 위치 일관화
- 자동 사라짐
- 같은 종류 연속 발생 시 중복 토스트 남발 금지

---

## 10-9. 모달 규칙

### 모달 사용 목적
유저가 반드시 멈춰서 판단해야 하는 경우

### 모달 사용 대상
- 매칭 시작 확인
- 작성 중 이탈 확인
- 로그아웃 확인
- 업로드 실패 재시도
- 매칭 실패 재시도

### 모달 구조 규칙
- 제목
- 짧은 설명
- 주 버튼
- 보조 버튼

예시:
- 제목: 작성 중인 내용이 사라질 수 있어요
- 본문: 나가시겠어요?

버튼:
- 계속 작성
- 나가기

### 모달 버튼 규칙
- 주 버튼은 오른쪽 또는 강조 스타일
- 파괴적 액션은 색상 / 문구 명확화
- 배경 터치로 닫기 허용 여부는 상황별 결정

추천:
- 이탈 확인, 로그아웃 확인은 배경 터치 닫기 가능
- 중요한 재시도 / 실패는 명시 버튼 클릭 우선

## 10-10. 바텀시트 규칙

### 사용 목적
짧은 선택지가 있을 때 별도 페이지보다 가볍게 처리

### 사용 대상
- 표현 남기기 선택

### 구조 규칙
- 제목
- 짧은 설명
- 선택지 리스트
- 닫기 / 완료 버튼

### 선택 상태 규칙
- 선택된 항목은 강조
- 항목 탭만으로는 서버 저장하지 않음(로컬 상태)
- **`선택 완료`** 버튼 클릭 시에만 서버에 반영(`POST /result-cards/:id/expression-reaction`)

## 10-11. 입력폼 상태 규칙

### 텍스트 입력

#### 정상
- 기본 border

#### focus
- 강조 border + soft shadow

#### invalid
- 붉은 border 또는 에러 텍스트

#### disabled
- 회색 처리

### 글자수 표시

대상:
- 한 줄 설명(미션 캡션): MVP **최대 60자**
- 한 줄 프로필: MVP **최대 40자**
- 소개(선택): MVP **최대 120자**

규칙:
- 현재 길이 / 최대 길이 표시
- 최대 초과 금지
- 최대 근접 시 색상 강조 가능

### 필수 입력 누락
- 제출 / 저장 버튼 `disabled`
- 필요하면 필드 아래 짧은 힌트

## 10-12. 리스트 / 카드 상태 규칙

### Board 카드
- 이미지 없을 경우 기본 placeholder
- `compare_line` 없을 경우 fallback 문구
- 반응 수 0이면 `반응 0` 또는 숨김 중 하나로 통일

추천:
- MVP에서는 반응 0도 표시 가능

### 결과 카드
- 두 카드 모두 동일 레이아웃 유지
- 한쪽 캡션이 짧아도 카드 높이 일관성 우선
- 이미지 비율 통일

### World 리스트
- 국가별 항목은 같은 높이
- 카운트 오른쪽 정렬
- 국가 설명이 길면 2줄 제한

## 10-13. 네비게이션 상태 규칙

### 하단 탭
- 현재 탭은 active 색상
- 비활성 탭은 muted 색상
- 라벨 길이 통일

현재 구조:
- Home
- Board
- World
- My

### 상단 아이콘
- 알림 아이콘 배지 수는 선택적
- 결과 보기 화면 등에서는 불필요한 상단 액션 제거

원칙:
- 상단 액션은 꼭 필요한 것만

## 10-14. 상태 전이 시 UI 갱신 규칙

### 예 1. 매칭 시작 후
- 홈 카드 버튼 즉시 `매칭 중...`
- 서버 응답 실패 시 원복

### 예 2. 제출 완료 후
- 미션 참여 화면 닫힘
- 오늘의 팀 미션 화면에서 진행 상태 갱신
- 버튼 `결과 보기` 비활성으로 변경
- 토스트 노출

### 예 3. 상대 제출 완료 알림 후
오늘의 팀 미션 화면 진입 시:
- 진행률 `2/2`
- `결과 보기` 활성

### 예 4. 카드 저장 후
- 저장 아이콘 즉시 on
- 실패 시 off 롤백

## 10-15. 접근성 / 글로벌 기준 최소 규칙

### 색만으로 상태 구분 금지
예:
- 완료 = 초록 칩 + 텍스트
- 대기 = 노랑 칩 + 텍스트

즉 색 + 문구 동시 제공

### 길이 제한 고려
다국어 확장 시 텍스트가 늘어날 수 있으므로:

- 버튼 문구는 짧게
- 섹션 제목도 짧게
- 긴 설명은 2~3줄 이내 제한

### 터치 영역
- 버튼 / 칩 / 아이콘 최소 터치 영역 확보
- 프로필 편집, 저장, 알림 등 작은 액션도 충분히 크게

## 10-16. 구현 체크리스트
각 화면 구현 시 아래를 체크해야 한다.

### 체크리스트
- [ ] 각 버튼이 `active` / `disabled` / `loading` 3상태를 가지는가
- [ ] 같은 유형의 성공 피드백이 토스트로 통일되어 있는가
- [ ] 확인이 필요한 동작만 모달을 쓰는가
- [ ] empty state와 error state가 구분되는가
- [ ] skeleton / 로딩이 레이아웃을 크게 흔들지 않는가
- [ ] 실패 시 UI 롤백 또는 상태 재조회가 정의되어 있는가
- [ ] 텍스트 길이가 길어져도 레이아웃이 깨지지 않는가
- [ ] 클릭 가능한 모든 요소의 disabled 조건이 명시되어 있는가

## 10-17. 최종 요약
이 앱의 UI 상태 규칙은 아래 5줄로 요약된다.

- 버튼은 `active / disabled / loading` 3상태만 사용한다.
- empty state와 error state는 반드시 구분한다.
- 성공은 토스트, 확인은 모달, 선택은 바텀시트로 통일한다.
- 상태가 바뀌면 버튼 / 문구 / 진행률이 즉시 같은 규칙으로 갱신되어야 한다.
- 모든 화면은 글로벌 확장을 고려해 짧고 일관된 상태 표현을 사용한다.
