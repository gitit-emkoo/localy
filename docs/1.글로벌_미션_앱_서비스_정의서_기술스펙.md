# 글로벌 미션 앱 서비스 정의서 / 기술 스펙

## 제품명
앱 표시 이름: **Localy** (구현·스토어·UI 문자열에 동일 적용)

## 1-1. 서비스 한 줄 정의
이 앱은 서로 다른 국가의 두 사용자가 하루 1개의 공통 미션에 참여하고(사진 1장과 한 줄 설명을 업로드) 결과 카드로 비교/반응하는 글로벌 공통 미션 수행 앱

## 1-2. 서비스 목적
이 서비스의 목적은 단순한 채팅이나 언어교환이 아닌,  
국가가 다른 사람들이 같은 질문에 각자의 이미지와 한 줄 설명으로 답하도록 만들고,  
그 결과를 비교하며 공감과 호기심을 유도하는 활동형 SNS라고 이해하면 된다.

핵심 가치는 4가지다.

- 글로벌 연결감: 다른 국가 사용자와 실제로 연결되어 같은 미션을 수행
- 낮은 진입장벽: 사진 1장 + 한 줄 설명만으로 참여 가능, 글로벌 장벽을 없앤 단순한 참여
- 비교 중심 경험: 같은 질문에 대한 다른 문화권의 답을 직관적으로 소비, 타 국가의 현장감 있는 문화를 확인
- 반응 중심 확장성: 결과 카드를 빠른 반응과 선택형 표현으로 소비

## 1-3. MVP 제품 원칙
MVP에서는 기능을 넓게 벌리지 않고, 아래 원칙을 따른다.

- 하루 공통 미션은 1개만 운영 (나중에 확장할 계획)
- 매칭은 2인 1팀, 서로 다른 국가 사용자 기준
- 제출물은 사진 1장 + 한 줄 설명만 허용됨
- 결과 소비는 결과 카드 단위
- 반응은 아이콘형 공통 반응(바로 선택) + 선택형 표현(바텀) 반응
- 홈은 단순하게 유지하고, 글로벌 현황은 별도 World 탭으로 분리
- 자유댓글, 장문 콘텐츠, 복잡한 랭킹은 MVP에서 제외함

## 1-4. 핵심 사용자 경험
사용자는 앱에 들어와 아래 흐름으로 움직인다.

1. 회원가입 / 로그인
2. 국가 및 닉네임, 나이, 프로필, 관심사 설정
3. 홈에서 오늘의 미션 확인
4. 매칭 시작으로 팀 매칭 요청
5. 매칭 완료 푸시 수신 또는 앱 내 상태 확인
6. 오늘의 팀 미션 화면 진입
7. 미션 참여 화면에서 사진 1장 + 한 줄 설명 제출
8. 상대 팀원이 제출 완료하면 결과 보기 가능
9. 결과 카드에서 반응 남기기
10. Board(타팀들의 결과 모음으로 글로벌 피드 개념)에서 다른 팀 결과 카드 탐색
11. My에서 내 결과 카드 / 저장한 카드 관리

## 1-7. 상태 모델
가장 먼저 이해해야 하는 건 상태 전이이다.

### 매칭 상태
- `idle`: 아직 매칭 시작 전
- `matching`: 매칭 대기 중
- `matched`: 매칭 완료

### 팀 미션 상태 (단일 기준 · 4분기)
코드·API·UI·DB·문서는 아래 네 값만 사용한다. (과거 3단계 표현은 사용하지 않음)

- `not_started`: 나·상대 모두 아직 제출 전
- `self_submitted_waiting`: 나는 제출 완료, 상대 미제출
- `peer_submitted_waiting`: 상대는 제출 완료, 나 미제출
- `ready_to_view`: 둘 다 제출 완료, 결과 보기 가능

### 결과 카드 상태
- `locked`: 아직 둘 다 제출되지 않아 결과 비공개
- `open`: 둘 다 제출 완료, 결과 공개

이 상태 정의를 명확히 이해하고 UI와 백엔드가 꼬이면 안 된다.

## 1-8. 권장 기술 스택
여기서는 글로벌 모바일 앱 MVP를 가장 빠르고 안정적으로 구현할 수 있는 조합으로 정리한다.

### 프론트엔드 앱
**Expo + React Native + TypeScript + Expo Router**

Expo는 하나의 JavaScript/TypeScript 프로젝트로 iOS, Android, web까지 확장 가능한 구조를 제공하고,  
Expo Router는 파일 기반 라우팅으로 화면 구조를 명확하게 관리할 수 있다.

이 앱은 상태 전이가 중요하므로, TypeScript는 필수로 두는 게 맞다.

### 상태 관리
**Zustand 또는 Redux Toolkit 중 하나, MVP는 Zustand 우선 추천**

이 앱은 전역 상태가 필요하다.

- 로그인 사용자
- 오늘의 미션
- 매칭 상태
- 팀 상태
- 알림 배지 수
- 저장 카드 수

MVP에선 구조가 복잡하지 않으므로, 보일러플레이트가 적은 Zustand로 진행한다.

### 서버 / 백엔드
**Supabase**

이 앱에는 다음이 필요하다.

- Auth
- Postgres DB
- Storage(이미지)
- Realtime(결과/반응/알림 동기화 일부)
- Edge Functions(매칭, 결과카드 생성, 푸시 트리거)

이 앱은 MVP에서 빠르게 구현해야 하므로,  
별도 백엔드 서버를 풀로 올리기보다 Supabase 중심 구조가 적합하다.

### 푸시 알림
**Firebase Cloud Messaging (FCM)**

이 앱은 알림이 핵심이다.  
특히 매칭 완료와 상대 제출 완료는 앱 재방문을 유도하는 핵심 이벤트다.

즉 푸시는 아래 조합으로 가는 것이 현실적이다.

- 클라이언트: Expo 앱
- 백엔드 트리거: Supabase Edge Functions
- 발송 채널: FCM / APNs

### 이미지 저장
**Supabase Storage**

미션 제출물은 사진 1장만 있으므로, Storage 요구가 단순하다.  
따라서 별도 S3 구성을 MVP부터 직접 운영하기보다 Supabase Storage가 더 적합하다.

### 인증
**Supabase Auth**

MVP에서는 이메일 또는 소셜 로그인 1~2개만 우선 붙이면 충분하다.  
Auth를 별도 구축하지 않고 Supabase Auth를 쓰는 것이 구현 속도와 유지보수 측면에서 유리하다.

### 분석
**PostHog 또는 Firebase Analytics 중 하나**

MVP에선 최소 이벤트만 수집하면 된다.

추천 이벤트:
- 매칭 시작
- 매칭 완료
- 미션 제출 완료
- 결과 보기 진입
- 반응 클릭
- 저장 클릭
- 에러 / 로그

### 에러 추적
**Sentry**

글로벌 앱은 디바이스 환경이 다양하므로 런타임 에러 수집이 중요하다.

## 1-10. 매칭 로직 원칙
MVP에서 매칭은 복잡한 추천 알고리즘이 아니라 조건형 랜덤 매칭으로 간다.

필수 조건:
- 서로 다른 국가
- 당일 공통 미션 기준
- 시간 민감형 미션인 경우 시차 제한
- 관심사 일부 교집합 우선
- 중복 매칭 최소화

이 앱은 하루 1개 공통 미션이라도, “지금 있는 장소” 같은 시간 민감형 질문은  
시차 차이가 큰 국가끼리 붙이면 UX가 어색할 수 있다.  
그래서 시간 민감형 미션은 시차 제한 규칙이 필요하다.

## 1-11. 알림 원칙
MVP 알림 이벤트는 이미 확정한 대로 아래 4개다.

- 매칭 완료
- 상대 제출 완료로 결과 보기 가능
- 새 반응 도착 알림은 묶음형 추천
- 오늘의 새 미션 오픈 알림은 선택사항 / 후순위

이 알림은 푸시와 인앱 알림 기록이 같이 가야 한다.

## 1-13-1. MVP 고정 파라미터 (합의본 · 임의 변경 금지)
아래 값은 MVP 기준으로 고정한다. 변경이 필요하면 **사전 합의 후** 문서·구현을 함께 수정한다.

### 텍스트 길이
- 미션 한 줄 캡션(`caption`): **최대 60자**
- 프로필 한 줄(`short_bio`): **최대 40자**
- 프로필 소개(`intro`, 선택): **최대 120자**

### 이미지 업로드
- 업로드 전 **클라이언트 압축 적용**
- 권장: **긴 변 1600px 이하**, 최종 파일 **2MB 이하** (초과 시 클라이언트에서 자동 압축 후 업로드)
- 허용 포맷: **jpg, jpeg, png** — **HEIC**는 가능하면 앱에서 **JPEG 변환 후** 업로드

### Board
- 목록 기본 정렬: **`latest`(최신순)**. `popular`(인기순)은 옵션으로 제공 가능

### 공유 · 딥링크 (MVP)
- **링크 공유 우선**, 이미지 공유는 후순위
- 결과 카드 상세로 직접 진입하는 딥링크 지원  
  - 예: `app://result-card/{resultCardId}`  
  - 예: `https://your-domain/result-card/{resultCardId}` (`your-domain`은 출시 도메인으로 확정 시 치환)

### 앱 UI 언어 (MVP 고정 4개)
- `ko`, `en`, `ja`, `pt-BR`
- 위 4개 외 지역·설정 조합은 **`en`을 fallback**으로 사용
- UI 문구·reaction 라벨·알림·버튼 등은 **전부 i18n 테이블** (하드코딩 금지)
- 번역 우선 언어(`translation_language`) 기본값은 **사용자 앱 언어** 기준
- 유저 생성 콘텐츠는 UI 언어와 별개로 **번역 가능한 구조** 유지

## 1-15. 최종 기술 스택 추천안
한 줄 추천안으로 정리하면 아래와 같다.

**Expo + React Native + TypeScript + Expo Router + Zustand + Supabase(Auth/Postgres/Storage/Realtime/Edge Functions) + Firebase Cloud Messaging + Sentry + Analytics**

이 조합이 현재 이 앱의 글로벌 기준 MVP에 가장 적합하다.  
이유는 구현 속도, 확장성, 모바일 최적화, 푸시 / 이미지 / 인증 / DB를 한 번에 다루는 현실성을 동시에 만족하기 때문이다.
