# API / 이벤트 정의서

## 문서 목적
이 문서는 아래를 명확히 정의한다.

- 어떤 버튼이 어떤 API를 호출하는지
- 어떤 API가 어떤 DB를 바꾸는지
- 어떤 서버 이벤트가 언제 발생하는지
- 어떤 상태가 언제 갱신되는지
- 어떤 알림 / 토스트 / 모달이 언제 뜨는지
- 어떤 동작은 클라이언트에서 하고, 어떤 건 서버에서 해야 하는지

이 문서의 핵심 목적은 **“화면은 맞는데 동작이 꼬이는 문제”를 막는 것**이다.

---

## 1. 문서 목적
이 문서는 MVP 앱의 모든 핵심 동작을 다음 기준으로 정의한다.

- 유저 액션
- 클라이언트 동작
- API 호출
- DB 변경
- 서버 후속 이벤트
- 클라이언트 상태 갱신
- 알림 / 토스트 / UI 피드백

이 문서는 프론트엔드, 백엔드, 코딩 AI 모두가 공유하는 실행 명세서다.

## 1-2. 설계 원칙

### 원칙 1. 클라이언트는 “요청”만 하고, 상태 확정은 서버가 한다
예:
- 매칭 시작 버튼 클릭 → 클라이언트가 팀을 직접 만들지 않음
- 제출 완료 → 클라이언트가 결과 카드 `open` 처리하지 않음

즉:
- 클라이언트 = 요청
- 서버 = 상태 확정

### 원칙 2. 상태 변화는 단일 책임으로 처리
예:
- 매칭 상태 변화는 매칭 로직만 처리
- 결과 카드 공개는 제출 완료 체크 로직만 처리
- 푸시 발송은 이벤트 트리거 로직만 처리

### 원칙 3. 화면은 API 결과를 렌더링할 뿐, 임의 상태를 만들지 않는다
즉 프론트에서 아래 같은 추정 로직은 금지한다.

- “상대도 했겠지”
- “결과 열어도 되겠지”

### 원칙 4. 모든 API는 최소한 아래를 정의한다
- 요청 목적
- 요청자 권한
- 요청 payload
- 응답 형태
- 실패 케이스
- 후속 이벤트

## 1-3. 엔드포인트 설계 스타일
MVP 기준 추천 방식:

- REST 스타일 기본
- 인증은 bearer token 기반
- 서버 로직은 Supabase Edge Functions 또는 API layer에서 처리
- DB 직접 호출이 아니라 도메인 단위 API로 캡슐화

예:
- `/match/start`
- `/missions/today`
- `/teams/:teamId`
- `/submissions`
- `/result-cards/:id/react`

## 1-5. 인증 / 세션 API

### 1) 이메일 가입 / 로그인

#### 목적
이메일 기반 가입 또는 로그인 처리

#### API
`POST /auth/email`

#### 요청
```json
{
  "email": "user@example.com"
}
```

#### 응답
```json
{
  "success": true,
  "authUserId": "uuid",
  "isNewUser": true,
  "accessToken": "jwt-token"
}
```

#### 후속 동작
- `isNewUser = true` → 프로필 설정 화면 이동
- `isNewUser = false` → 프로필 완료 여부 조회 후 이동

#### 실패
- 이메일 형식 오류
- 인증 실패
- 네트워크 오류

### 2) 세션 확인

#### API
`GET /auth/session`

#### 목적
앱 시작 시 로그인 여부와 유저 기본 상태 확인

#### 응답
```json
{
  "signedIn": true,
  "userId": "uuid",
  "isProfileCompleted": true
}
```

#### 후속 동작
- `signedOut` → 로그인 화면
- `signedIn + incomplete` → 프로필 설정
- `signedIn + completed` → 홈

## 1-8. 매칭 API

### 7) 매칭 시작

#### API
`POST /match/start`

#### 목적
오늘 미션 기준 매칭 요청 생성

#### 요청
```json
{
  "missionId": "mission-uuid"
}
```

#### 응답
```json
{
  "success": true,
  "matchRequestId": "uuid",
  "status": "matching"
}
```

#### DB 변경
- `match_requests` row 생성 또는 갱신
- `status = matching`

#### 후속 서버 이벤트
- 매칭 로직 큐 / 함수 실행
- 조건 맞는 상대 탐색

#### 클라이언트 후속 동작
- 홈 오늘의 미션 카드 상태 `matching`
- 버튼 `매칭 중...` 비활성

#### 실패
- 이미 활성 매칭 요청 있음
- 오늘 미션 없음
- 프로필 미완성
- 서버 오류

### 8) 매칭 상태 조회

#### API
`GET /match/status?missionId=...`

#### 목적
홈에서 현재 내 매칭 상태 확인

#### 응답 예시

매칭 전:
```json
{
  "status": "idle"
}
```

매칭 중:
```json
{
  "status": "matching",
  "matchRequestId": "uuid"
}
```

매칭 완료:
```json
{
  "status": "matched",
  "matchRequestId": "uuid",
  "teamId": "team-uuid",
  "peerCountryCode": "JP",
  "peerCountryName": "Japan"
}
```

#### 클라이언트 후속 동작
- 홈 버튼 상태 갱신

### 9) 매칭 취소(선택)
MVP에서 필수는 아니지만, 넣을 경우:

#### API
`POST /match/cancel`

#### 조건
- 상태가 `matching`일 때만 가능

### 매칭 완료 서버 이벤트

#### 이벤트명
`MATCH_COMPLETED`

#### 트리거 조건
- 조건에 맞는 2명의 요청이 매칭됨
- `teams` 생성 완료
- 관련 `match_requests.status = matched`

#### 서버 후속 작업
- `teams` row 생성
- 두 유저 `match_requests.team_id` 연결
- 알림 row 생성
- 푸시 발송

## 1-10. 제출 API

### 11) 미션 제출

#### API
`POST /submissions`

#### 목적
사진 1장 + 한 줄 설명 제출

#### 요청
```json
{
  "teamId": "team-uuid",
  "missionId": "mission-uuid",
  "photoUrl": "https://storage/...",
  "captionOriginal": "퇴근 전에 카페에서 잠깐 정리 중이에요."
}
```

#### 응답
```json
{
  "success": true,
  "submissionId": "submission-uuid",
  "teamMissionStatus": "self_submitted_waiting"
}
```

#### DB 변경
- `submissions` insert

#### 서버 후속 로직
같은 팀의 다른 submission 존재 여부 조회

없으면:
- 팀 상태를 waiting 상태로 계산

있으면:
- `result_cards` 생성 또는 `open` 처리
- `teams.match_status` 갱신
- 상대 유저에게 결과 보기 가능 알림 생성 / 발송

#### 클라이언트 후속 동작
- 토스트: `제출이 완료되었어요`
- 오늘의 팀 미션 화면 복귀
- 팀 상태 재조회

#### 실패
- team 권한 없음
- 이미 제출 완료
- 이미지 누락
- caption 누락
- storage 업로드 실패
- 서버 오류

### 제출 완료 서버 이벤트

#### 이벤트명
`SUBMISSION_CREATED`

#### 트리거 조건
- `submissions` insert 성공

#### 후속 분기
경우 A. 상대 제출 없음
- 팀 미션 상태 = waiting

경우 B. 상대 제출 있음
- `RESULT_CARD_READY` 이벤트 발생

- `result_cards` 생성 또는 `status = open`

- `peer_submitted` 알림 생성
- 푸시 발송
- Board / World 집계 포함 가능 상태로 전환

## 1-12. 반응 API

### 13) 공통 아이콘 반응 등록 / 변경

#### API
`POST /result-cards/:id/icon-reaction`

#### 요청
```json
{
  "reactionType": "interesting"
}
```

#### 정책 (MVP 고정)
- 유저당 카드당 **하나의 아이콘 반응**만 존재
- 요청 `reactionType`이 **현재와 같으면** 반응 **취소**(행 삭제, 집계 감소)
- **다르면** 기존 반응을 **새 타입으로 변경**(덮어쓰기, 집계 이동)
- 반응이 없을 때는 **신규 생성**

#### 응답
```json
{
  "success": true,
  "myReaction": "interesting",
  "iconReactionSummary": {
    "interesting": 129,
    "nice": 94,
    "surprised": 76,
    "relatable": 51
  }
}
```

#### 클라이언트 후속 동작
- 숫자 즉시 갱신
- 필요시 토스트 생략 가능

### 14) 아이콘 반응 취소(선택 · 대안 API)

MVP에서는 **13) `POST`의 토글 동작**(동일 타입 재요청 = 취소)으로 충분하다.  
별도 `DELETE`만으로 취소하는 방식은 구현 편의상 선택 가능하나, **문서상 단일 동작은 13)에 정의한 규칙**을 따른다.

#### API (선택)
`DELETE /result-cards/:id/icon-reaction`

### 15) 표현 반응 등록 / 변경

#### API
`POST /result-cards/:id/expression-reaction`

#### 클라이언트 동작 (MVP 고정)
- 바텀시트에서 항목을 탭할 때는 **로컬 상태만** 변경
- 사용자가 **`선택 완료`** 버튼을 눌렀을 때만 아래 `POST` 호출(서버 반영)

#### 요청
```json
{
  "expressionKey": "want_to_visit"
}
```

#### 서버 동작
- category에 맞는 expression text resolve
- 유저당 카드당 1개 upsert

#### 응답
```json
{
  "success": true,
  "myExpressionReaction": {
    "expressionKey": "want_to_visit",
    "expressionText": "가보고 싶어요"
  },
  "expressionSummary": [
    { "expressionKey": "want_to_visit", "expressionText": "가보고 싶어요", "count": 13 },
    { "expressionKey": "similar", "expressionText": "우리랑 비슷하네요", "count": 9 }
  ]
}
```

#### 클라이언트 후속 동작
- 바텀시트 닫기
- 토스트: `표현을 남겼어요`
- 요약 목록 갱신

## 1-13. Board API

### 16) Board 목록 조회

#### API
`GET /board/cards?sort=...&cursor=...`

#### 쿼리 파라미터 (MVP)
- `sort`: 정렬 방식 — **생략 시 기본값 `latest`**
  - **`latest`**: 최신순(예: 결과 카드 공개 시각·생성 시각 기준 내림차순)
  - **`popular`**: 인기순(예: 반응 수·집계 합 기준 내림차순, 동률 시 `latest` 보조)
- `cursor`: 페이지네이션 커서(기존과 동일)

추후 필터·정렬 키를 늘릴 때는 같은 엔드포인트에 옵션을 확장한다.

#### 목적
Board 그리드 / 리스트 데이터 조회

#### 응답
```json
{
  "items": [
    {
      "resultCardId": "uuid",
      "countryPairLabel": "Korea × Thailand",
      "thumbnailUrl": "https://...",
      "compareLine": "서울의 카페 vs 방콕의 야시장",
      "reactionCount": 143
    }
  ],
  "nextCursor": "opaque-cursor"
}
```

#### 규칙
- `open` 카드만 반환
- MVP에서는 **`sort=latest` / `sort=popular`** 두 가지를 지원하면 됨(홈 Board 미리보기는 동일 API의 축약 호출로 맞출 수 있음)

## 1-21. 클라이언트 캐시 / 재조회 전략
최적화와 안정성을 위해 다음 원칙을 둔다.

### 즉시 갱신 + 서버 재검증
반응 / 저장 같은 가벼운 액션은 optimistic update 가능  
하지만 최종값은 서버 응답 기준으로 맞춘다.

### 반드시 서버 재조회가 필요한 곳
- 매칭 완료 후 홈
- 제출 완료 후 오늘의 팀 미션
- 결과 보기 진입 시 카드 상세
- 알림 페이지

### 캐시 가능
- 내 프로필
- 설정
- 오늘의 미션
- Board 첫 페이지 일부

## 1-23. 절대 헷갈리면 안 되는 핵심

### 핵심 1
결과 카드 생성 / 공개는 서버만 한다  
프론트에서 둘 다 제출됐다고 판단해서 임의로 카드 `open` 처리 금지

### 핵심 2
결과 보기 화면은 하나만 사용한다  
내 팀 결과 / Board 결과 모두 동일 화면 사용

### 핵심 3
매칭 시작과 매칭 완료는 다른 이벤트다
- 버튼 클릭 = 요청 시작
- 실제 매칭 성사 = 서버 이벤트

### 핵심 4
제출 성공은 곧 결과 보기 가능이 아니다  
상대 제출 여부를 서버가 다시 판단해야 함

### 핵심 5
표현 반응은 자유댓글이 아니다  
선택형 문구만 저장

## 1-24. 최종 요약
이 앱의 API / 이벤트 구조는 아래 6줄로 요약된다.

- 유저는 요청만 보내고, 상태 확정은 서버가 한다.
- 홈의 매칭 상태는 `match_requests` 기반이다.
- 팀 미션 상태는 `teams + submissions` 기반이다.
- 결과 카드는 두 제출 완료 후 서버가 공개한다.
- 결과 카드 상세는 내 팀 / 타 팀 공통 화면 하나로 처리한다.
- 반응 / 저장 / 알림은 모두 중복 방지와 상태 일관성을 우선한다.
