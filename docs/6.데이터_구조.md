# 데이터 구조

## 문서 목적
이 문서는 아래를 완벽하게 바로 이해할 수 있게 만드는 게 목적이다.

- 어떤 테이블 / 컬렉션이 필요한지
- 각 데이터가 무슨 역할인지
- 어떤 테이블끼리 연결되는지
- 어떤 값이 상태를 결정하는지
- MVP에서 어디까지 저장하고, 무엇은 나중에 확장할지

중요한 원칙은 하나다.

**MVP는 단순하게, 하지만 확장 가능하게.**

즉 처음부터 너무 많은 테이블을 만들지 않되,  
나중에 1차 정식버전에서 기능이 커져도 구조를 갈아엎지 않게 해야 한다.

# 6. 데이터 구조 초안

## 6-1. 데이터 구조 설계 원칙
이 앱의 데이터 구조는 아래 원칙으로 설계한다.

- 하루 1개 공통 미션 구조를 기준으로 설계한다.
- 매칭팀(2인 1팀)을 핵심 단위로 둔다.
- 제출물은 반드시 사진 1장 + 한 줄 설명 구조로 고정한다.
- 결과 카드는 매칭된 두 유저의 제출물을 묶은 결과 단위로 관리한다.
- 반응은 아이콘 반응과 표현 반응을 분리한다.
- 저장(즐겨찾기), 알림, 설정은 독립 테이블로 둔다.
- 상태 판별은 프론트 임의 계산보다 DB 필드와 서버 계산 결과를 우선한다.

## 6-2. 전체 엔티티 목록
MVP 기준 핵심 엔티티는 아래 11개다.

- `users`
- `user_profiles`
- `user_interests`
- `missions`
- `match_requests`
- `teams`
- `submissions`
- `result_cards`
- `card_icon_reactions`
- `card_expression_reactions`
- `saved_cards`
- `notifications`
- `user_settings`

실제로는 `users`와 `auth.users`를 분리해서 보는 게 좋다.  
Supabase Auth를 쓸 경우 인증 계정은 Supabase 쪽에서 관리하고, 앱에서 쓰는 프로필 정보는 별도 프로필 테이블로 둔다.

## 6-3. 테이블별 정의

### 1) users / auth.users

#### 역할
인증 계정 원본

#### 설명
- 이메일 가입 / 로그인 정보
- 비밀번호 / 인증 관련 정보
- 앱 도메인 데이터와 분리

#### 주의
앱 로직에서 직접 사용하는 프로필 정보는 `user_profiles`에 둔다.

---

### 2) user_profiles

#### 역할
앱에서 실제로 사용하는 유저 프로필 정보

#### 주요 필드
- `id`: UUID, PK
- `auth_user_id`: UUID, unique, `auth.users` 참조
- `nickname`: string
- `age`: integer or smallint
- `country_code`: string (`KR`, `JP`, `BR`)
- `country_name`: string (`Korea`, `Japan`)
- `short_bio`: string (예: `32세 · Founder`) — MVP **최대 40자**
- `intro`: text nullable — MVP **최대 120자** (선택 필드)
- `profile_image_url`: string nullable
- `is_profile_completed`: boolean
- `created_at`: timestamp
- `updated_at`: timestamp

#### 설명
- MVP에서는 닉네임, 나이, 국가, 짧은 프로필, 관심사만 필수
- `is_profile_completed`로 최초 설정 완료 여부를 판단

#### 비고
- `country_code`는 반드시 ISO 스타일로 고정
- 프론트 표시용 텍스트와 로직용 코드는 분리

---

### 3) user_interests

#### 역할
유저 관심사 다중 선택 저장

#### 주요 필드
- `id`: UUID, PK
- `user_id`: UUID, FK -> `user_profiles.id`
- `interest_key`: string

#### interest_key 예시
- `food`
- `daily_life`
- `place`
- `emotion`
- `study`
- `daily_spending`

#### 설명
- 관심사는 다중 선택
- 별도 배열 필드보다 정규화 테이블이 확장에 유리

#### 제약
- `(user_id, interest_key) unique`

---

### 4) missions

#### 역할
하루 1개 운영되는 공통 미션

#### 주요 필드
- `id`: UUID, PK
- `mission_date`: date
- `category_key`: string
- `title`: string
- `short_description`: string
- `notice_text`: text or string
- `is_time_sensitive`: boolean
- `status`: string (`draft`, `published`, `closed`)
- `created_at`: timestamp
- `updated_at`: timestamp

#### 설명
- MVP에서는 하루 1개만 `published` 상태
- `mission_date` 기준으로 당일 미션을 찾음
- `is_time_sensitive`는 시차 기반 매칭 규칙에 사용

#### category_key 예시
- `place`
- `food`
- `daily_life`
- `emotion`
- `study`
- `daily_spending`

---

### 5) match_requests

#### 역할
유저가 오늘 매칭을 시작했는지 기록하는 요청 단위

#### 주요 필드
- `id`: UUID, PK
- `user_id`: UUID, FK -> `user_profiles.id`
- `mission_id`: UUID, FK -> `missions.id`
- `status`: string (`idle`, `matching`, `matched`, `failed`, `cancelled`)
- `timezone_offset_minutes`: integer
- `preferred_interest_keys`: jsonb nullable
- `team_id`: UUID nullable, FK -> `teams.id`
- `requested_at`: timestamp
- `matched_at`: timestamp nullable
- `failed_at`: timestamp nullable

#### 설명
- 유저가 매칭 시작 누르면 row 생성
- 매칭 완료되면 `team_id` 연결
- 하루 / 미션 기준 중복 요청 방지 필요

#### 제약
- 동일 `user_id + mission_id`에 대해 활성 요청 1개만 허용

---

### 6) teams

#### 역할
당일 미션에 대해 2명의 유저를 묶은 팀

#### 주요 필드
- `id`: UUID, PK
- `mission_id`: UUID, FK -> `missions.id`
- `user_a_id`: UUID, FK -> `user_profiles.id`
- `user_b_id`: UUID, FK -> `user_profiles.id`
- `user_a_country_code`: string
- `user_b_country_code`: string
- `match_status`: string (`matched`, `ready_to_view`, `closed`)
- `created_at`: timestamp
- `updated_at`: timestamp

#### 설명
- 팀은 무조건 2명
- 같은 국가끼리는 절대 금지
- 앱의 본질은 나와 다른 국가 유저와 팀이 되어 미션하는 것이기 때문
- 한 팀은 하나의 미션에만 속함

#### 제약
- `user_a_id != user_b_id`
- `user_a_country_code != user_b_country_code`

#### 비고
국가 코드를 팀에 중복 저장하는 이유는 조회 최적화와 카드 렌더링 편의성 때문이다.

---

### 7) submissions

#### 역할
팀원 각자의 제출물

#### 주요 필드
- `id`: UUID, PK
- `team_id`: UUID, FK -> `teams.id`
- `mission_id`: UUID, FK -> `missions.id`
- `user_id`: UUID, FK -> `user_profiles.id`
- `photo_url`: string
- `caption_original`: string — MVP **최대 60자**
- `caption_translated`: string nullable
- `submitted_at`: timestamp
- `status`: string (`submitted`, `deleted`)

#### 설명
- 팀당 유저 1명당 제출물 1개
- 사진 1장 + 한 줄 설명
- 번역 결과는 미리 저장하거나 조회 시 생성 가능

#### 제약
- `(team_id, user_id) unique`

#### 비고 (MVP 고정)
- 제출 후 **수정·삭제·재제출 불가**. 제출 시점 이후 제출물은 고정된다.

---

### 8) result_cards

#### 역할
두 제출물이 모두 완료되었을 때 생성되는 결과 카드

#### 주요 필드
- `id`: UUID, PK
- `mission_id`: UUID, FK -> `missions.id`
- `team_id`: UUID, FK -> `teams.id`
- `submission_a_id`: UUID, FK -> `submissions.id`
- `submission_b_id`: UUID, FK -> `submissions.id`
- `status`: string (`locked`, `open`)
- `compare_line`: string nullable
- `total_icon_reaction_count`: integer default 0
- `total_expression_reaction_count`: integer default 0
- `created_at`: timestamp
- `opened_at`: timestamp nullable

#### 설명
- 두 제출물이 모두 생기면 생성 또는 `open` 전환
- Board와 결과 보기의 핵심 객체
- 반응 수는 캐시 필드로 유지 가능

#### 상태 규칙
- `locked`: 둘 다 제출 완료 전, 외부 노출 금지
- `open`: 둘 다 제출 완료 후 공개

#### 비고
`compare_line`은 자동 생성 또는 운영 입력 가능  
MVP에서는 nullable로 두고 추후 붙여도 된다.

---

### 9) card_icon_reactions

#### 역할
공통 아이콘 반응 저장

#### 주요 필드
- `id`: UUID, PK
- `result_card_id`: UUID, FK -> `result_cards.id`
- `user_id`: UUID, FK -> `user_profiles.id`
- `reaction_type`: string
- `created_at`: timestamp
- `updated_at`: timestamp

#### reaction_type (고정 키 · UI는 i18n)
아래 네 값만 사용한다. 라벨·이모지 문자열은 코드 하드코딩하지 않고 **i18n 또는 reaction 설정 테이블**에서 관리한다.

- `interesting` — 기본 한글 예: 👀 흥미로워요
- `nice` — 기본 한글 예: ❤️ 좋아보여요
- `surprised` — 기본 한글 예: 😮 의외예요
- `relatable` — 기본 한글 예: 🌍 공감돼요

#### 설명
유저가 카드에 남긴 아이콘 반응

#### 제약(추천)
- `(result_card_id, user_id) unique`

#### 정책 (MVP 고정)
- 유저당 카드당 아이콘 반응은 **하나만** 존재
- **다른 아이콘** 선택 시 기존 행을 **새 `reaction_type`으로 변경**
- **같은 아이콘**을 다시 누르면 반응 **취소**(행 삭제)

---

### 10) card_expression_reactions

#### 역할
선택형 한 줄 표현 반응 저장

#### 주요 필드
- `id`: UUID, PK
- `result_card_id`: UUID, FK -> `result_cards.id`
- `user_id`: UUID, FK -> `user_profiles.id`
- `expression_key`: string
- `expression_text`: string
- `created_at`: timestamp
- `updated_at`: timestamp

#### 설명
- 자유댓글 아님
- 카테고리별 준비된 문구 중 하나를 선택
- 유저당 카드당 1개만 허용 추천

#### 제약
- `(result_card_id, user_id) unique`

#### 비고
`expression_key`와 `expression_text`를 같이 저장하면  
표현 문구가 나중에 바뀌어도 당시 선택 내용을 보존 가능하다.

---

### 11) saved_cards

#### 역할
다른 팀 결과 카드 즐겨찾기 저장

#### 주요 필드
- `id`: UUID, PK
- `user_id`: UUID, FK -> `user_profiles.id`
- `result_card_id`: UUID, FK -> `result_cards.id`
- `created_at`: timestamp

#### 제약
- `(user_id, result_card_id) unique`

#### 설명
`My > 저장한 카드 목록`에 사용

---

### 12) notifications

#### 역할
인앱 알림 기록 저장

#### 주요 필드
- `id`: UUID, PK
- `user_id`: UUID, FK -> `user_profiles.id`
- `notification_type`: string
- `title`: string
- `body`: string
- `target_type`: string nullable
- `target_id`: UUID nullable
- `is_read`: boolean default false
- `created_at`: timestamp
- `read_at`: timestamp nullable

#### notification_type 예시
- `match_completed`
- `peer_submitted`
- `new_reaction`
- `new_daily_mission`

#### target_type 예시
- `team`
- `result_card`
- `mission`

#### 설명
- 푸시와 별도로 인앱 알림 목록 유지
- 클릭 시 `target`으로 이동

---

### 13) user_settings

#### 역할
유저별 설정 저장

#### 주요 필드
- `id`: UUID, PK
- `user_id`: UUID, FK -> `user_profiles.id`
- `push_match_completed`: boolean default true
- `push_peer_submitted`: boolean default true
- `push_new_reaction`: boolean default true
- `push_new_daily_mission`: boolean default false
- `app_language`: string default `en` — MVP 허용 값: `ko` | `en` | `ja` | `pt-BR` (그 외 fallback은 `en`)
- `translation_language`: string default `en` — 기본은 **앱 UI 언어와 동일**을 권장
- `created_at`: timestamp
- `updated_at`: timestamp

#### 설명
- 알림 설정
- 언어 설정
- MVP에서 꼭 필요한 설정만 저장

## 6-5. 테이블 관계 요약

### 핵심 관계
- `auth.users` 1 : 1 `user_profiles`
- `user_profiles` 1 : N `user_interests`
- `missions` 1 : N `match_requests`
- `missions` 1 : N `teams`
- `teams` 1 : 2 `submissions`
- `teams` 1 : 1 `result_cards`
- `result_cards` 1 : N `card_icon_reactions`
- `result_cards` 1 : N `card_expression_reactions`
- `result_cards` 1 : N `saved_cards`
- `user_profiles` 1 : N `notifications`
- `user_profiles` 1 : 1 `user_settings`

## 6-6. 상태 계산 기준
이 앱은 상태가 매우 중요하므로, 어떤 상태를 어디서 계산하는지 고정해야 한다.

### 홈의 매칭 상태는 어디서 오나
기준:
- `match_requests.status`

즉 홈의 버튼 상태는 `match_requests`를 기준으로 판단한다.

### 오늘의 팀 미션 상태는 어디서 오나
기준:
- `teams`
- `submissions`

판단 규칙:
- 제출 없음 → `not_started`
- 내 것만 있음 → `self_submitted_waiting`
- 상대 것만 있음 → `peer_submitted_waiting`
- 둘 다 있음 → `ready_to_view`

### 결과 카드 공개 여부는 어디서 오나
기준:
- `result_cards.status`

`open`인 카드만 Board 노출

## 6-7. 대표 쿼리 기준

### 홈 로드 시 필요한 데이터
- 오늘의 `published mission`
- 내 최신 `match_request`
- `match_request`에 연결된 `team`
- `team`에 연결된 `submissions` 존재 여부
- Board 미리보기용 `open result_cards` 일부

### 오늘의 팀 미션 화면 로드 시 필요한 데이터
- 오늘 `mission`
- 내 `team`
- `team`의 두 유저 프로필
- 현재 `team`에 연결된 `submissions`
- 결과 카드 존재 여부

### 결과 보기 화면 로드 시 필요한 데이터
- `result_card`
- 두 `submission`
- 두 유저 프로필
- icon reaction summary
- expression reaction summary
- 내가 남긴 icon / expression reaction
- 내가 저장했는지 여부

### My 화면 로드 시 필요한 데이터
- 내 profile
- 내 활동 요약용 count
- `user_settings` 일부

## 6-8. 활동 요약 계산값
My 화면의 활동 요약 수치는 캐시 없이 쿼리로도 가능하지만,  
MVP에서 트래픽이 크지 않다면 우선 쿼리 기반으로 시작해도 된다.

예시:
- 참여 미션 수 = 내가 속한 `team` 중 제출 완료 건 수
- 받은 반응 수 = 내 제출물이 포함된 `result_card`의 reaction 합계
- 남긴 표현 수 = 내가 남긴 `card_expression_reactions` 개수

## 6-9. 파일 / 이미지 저장 원칙

### 사진 업로드 (MVP 고정)
- 업로드 전 **클라이언트 압축** 적용
- 권장: **긴 변 1600px 이하**, 최종 파일 **2MB 이하** (초과 시 클라이언트 자동 압축 후 업로드)
- 허용 포맷: **jpg, jpeg, png** — **HEIC**는 앱에서 **JPEG 변환 후** 업로드 권장
- Storage에는 위 조건을 만족한 파일만 올리고, DB에는 **`photo_url`만** 저장

### 주의
- DB에 base64 저장 금지
- 반드시 storage URL만 저장

## 6-10. 실수하기 쉬운 포인트
이 부분이 매우 중요하다.

### 주의 1
`results`라는 추상 이름보다 반드시 `result_cards`로 고정  
앱의 핵심 객체가 카드 단위라는 점을 코드에서도 반영해야 한다.

### 주의 2
매칭 상태와 팀 미션 상태를 섞지 말 것  
- 매칭 상태 = `match_requests`
- 팀 미션 상태 = `teams + submissions`

### 주의 3
내 결과 보기와 타 팀 결과 보기를 별도 테이블로 나누지 말 것  
같은 `result_cards`를 사용한다.

### 주의 4
표현 반응을 자유댓글처럼 구현하지 말 것  
반드시 선택형 문구만 저장한다.

### 주의 5
하루 공통 미션 1개 구조를 전제로 하되,  
테이블은 나중에 여러 미션으로 확장 가능한 형태를 유지한다.

## 6-11. 추천 타입 정의 예시

## 6-12. 최종 정리
이 데이터 구조 초안의 핵심은 아래 5줄로 요약된다.

- 유저 프로필과 관심사는 매칭 기준 데이터다.
- 미션은 하루 1개 공통 운영이지만, 테이블은 확장 가능하게 만든다.
- 팀(2인)과 제출물(사진 1장 + 한 줄)이 핵심 생산 데이터다.
- 결과 카드가 앱의 핵심 소비 객체다.
- 반응 / 저장 / 알림 / 설정은 모두 결과 카드와 유저 중심으로 분리 저장한다.
