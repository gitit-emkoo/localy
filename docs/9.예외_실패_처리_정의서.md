# 예외 / 실패 처리 정의서

## 문서 개요
이 문서는 MVP를 구현할 때 가장 많이 실수하는 부분, 즉  
**“정상 흐름은 되는데 실패 상황에서 앱이 깨지거나, 멈추거나, 상태가 꼬이는 문제”**를 막기 위한 문서다.

이 앱은 특히 아래 실패가 중요하다.

- 매칭 실패
- 업로드 실패
- 제출 실패
- 결과카드 미생성
- 반응 실패
- 저장 실패
- 푸시 / 알림 불일치
- 로그인 / 세션 문제
- 빈 데이터 상태

즉 이 문서는 오류 메시지 모음이 아니라,  
**실패했을 때 앱이 어떤 상태를 유지해야 하는지까지 정의하는 문서**다.

---

## 9-1. 문서 목적
이 문서는 앱 내 주요 기능에서 발생 가능한 실패 / 예외 상황을 정의하고,  
각 상황에 대해 아래를 명확히 정리한다.

- 어떤 상황에서 실패가 발생하는가
- 사용자에게 어떤 UI로 알려야 하는가
- 기존 상태를 유지할지 되돌릴지
- 재시도를 허용하는가
- 이후 이동 경로는 무엇인가

---

## 9-2. 예외 처리 기본 원칙

### 원칙 1. 실패는 조용히 숨기지 않는다
사용자가 인식해야 하는 실패는 반드시 알려야 한다.

예:
- 제출 실패
- 저장 실패
- 매칭 실패

### 원칙 2. 실패 후 상태는 일관되어야 한다
예:
- 저장 실패했는데 UI는 저장된 것처럼 보이면 안 됨
- 제출 실패했는데 홈에서 제출 완료처럼 보이면 안 됨

즉, 실패 시 **UI 롤백 또는 상태 유지**를 명확히 해야 한다.

### 원칙 3. 재시도 가능한 실패와 불가능한 실패를 구분한다
예:
- 네트워크 오류 → 재시도 가능
- 권한 없음 → 재시도보다 재로그인 / 이동 필요
- 이미 제출 완료 → 재시도 불가, 상태 재조회 필요

### 원칙 4. 실패 메시지는 짧고 행동 유도형으로 쓴다
좋은 예:
- 업로드에 실패했어요. 다시 시도해 주세요.
- 팀을 찾지 못했어요. 다시 매칭해 주세요.

나쁜 예:
- Unknown network exception 503

### 원칙 5. 글로벌 앱 기준으로 텍스트 의존도를 줄인다
가능하면 아래를 함께 쓴다.

- 짧은 문구
- 명확한 버튼
- 시각적 상태 변화

---

## 9-3. 실패 유형 분류
MVP 기준 실패는 아래 6개로 분류한다.

- 인증 / 세션 실패
- 매칭 실패
- 제출 / 업로드 실패
- 결과 카드 접근 실패
- 반응 / 저장 실패
- 데이터 없음 / empty state

---

## 9-4. 인증 / 세션 실패

### A. 로그인 실패

#### 발생 상황
- 이메일 형식 오류
- 인증 서버 실패
- 토큰 발급 실패

#### 사용자 표시
- 인라인 에러 또는 토스트

예:
- 이메일 형식을 확인해 주세요
- 로그인에 실패했어요. 다시 시도해 주세요

#### 상태 처리
- 로그인 화면 유지
- 기존 입력값 유지

#### 재시도
- 가능

### B. 세션 만료

#### 발생 상황
- 로그인 상태였지만 토큰 만료
- 백엔드 인증 실패

#### 사용자 표시
- 모달 또는 전체 화면 안내

예:
- 다시 로그인해 주세요

#### 상태 처리
- 로컬 인증 상태 제거
- 보호 화면 접근 차단
- 로그인 화면으로 강제 이동

#### 재시도
- 재로그인 필요

## 9-5. 초기 프로필 설정 실패

### A. 프로필 저장 실패

#### 발생 상황
- 필수값 누락
- 서버 저장 실패
- 네트워크 오류

#### 사용자 표시
- 토스트 또는 인라인 에러

예:
- 프로필 저장에 실패했어요
- 필수 항목을 입력해 주세요

#### 상태 처리
- 입력값 유지
- 화면 이탈 금지
- 시작하기 버튼 로딩 해제

#### 재시도
- 가능

---

## 9-6. 매칭 실패

### A. 매칭 시작 요청 실패

#### 발생 상황
- 네트워크 오류
- 오늘 미션 없음
- 이미 매칭 요청 존재
- 서버 오류

#### 사용자 표시
- 모달 또는 토스트

예:
- 매칭을 시작하지 못했어요
- 다시 시도해 주세요

#### 상태 처리
- 홈의 오늘의 미션 카드는 `idle` 유지
- 버튼은 다시 `매칭 시작`

#### 재시도
- 가능

### B. 매칭 대기 중 오래 걸림

#### 발생 상황
- 사용자가 적어서 즉시 매칭되지 않음

#### 사용자 표시
- 홈 상태 문구 유지

예:
- 비슷한 시간대의 다른 국가 팀원을 찾고 있어요

필요 시 안내 문구:
- 앱을 닫아도 매칭되면 알려드릴게요

#### 상태 처리
- `matching` 유지
- 버튼 비활성 유지

#### 재시도
- 별도 재시도 버튼 없음
- 시스템이 계속 탐색

### C. 매칭 실패 확정

#### 발생 상황
- 일정 시간 내 매칭 불가
- 서버가 실패로 종료

#### 사용자 표시
- 상태 문구 변경
- 버튼 `다시 매칭하기`

예:
- 팀을 찾지 못했어요. 다시 시도해 주세요

#### 상태 처리
- 홈 상태 `match_failed`
- 오늘의 미션 참여 진입 금지

#### 재시도
- 가능

### D. 매칭 완료 푸시 발송 실패

#### 발생 상황
- 팀은 생성됐지만 푸시가 발송되지 않음

#### 사용자 표시
- 별도 즉시 표시 없음

#### 상태 처리
- 앱 재진입 시 `/match/status` 조회로 보완
- 홈에서는 `matched` 정상 표시

#### 재시도
- 푸시 발송은 서버 재시도 가능
- 클라이언트는 상태 조회로 복구

---

## 9-7. 미션 참여 / 업로드 실패

### A. 사진 업로드 실패

#### 발생 상황
- 이미지 선택 후 storage 업로드 실패
- 파일 용량 초과
- 네트워크 오류

#### 사용자 표시
- 모달 또는 토스트

예:
- 사진 업로드에 실패했어요
- 다시 시도해 주세요

#### 상태 처리
- 업로드 전 상태 유지
- 기존 선택 이미지는 유지 가능하면 유지
- 제출 버튼 비활성

#### 재시도
- 가능

### B. 제출 실패

#### 발생 상황
- `photoUrl` 누락
- `caption` 누락
- API 실패
- 서버 오류

#### 사용자 표시
- 모달

예:
- 제출에 실패했어요
- 다시 시도해 주세요

#### 상태 처리
- 미션 참여 화면 유지
- 입력값 유지
- 버튼 로딩 해제

#### 재시도
- 가능

### C. 작성 중 이탈

#### 발생 상황
- 사진 또는 설명 입력 후 뒤로가기
- 제출 전

#### 사용자 표시
- 모달

예:
- 작성 중인 내용이 사라질 수 있어요
- 나가시겠어요?

#### 상태 처리
- 유저가 취소하면 그대로 유지
- 유저가 나가기를 선택하면 이전 화면 이동

### D. 중복 제출 시도

#### 발생 상황
- 이미 제출 완료된 유저가 다시 제출 요청
- 중복 탭
- 느린 네트워크로 재전송

#### 사용자 표시
- 토스트 또는 모달

예:
- 이미 제출이 완료되었어요

#### 상태 처리
- 팀 미션 화면 상태 재조회
- `self_submitted_waiting` 또는 `ready_to_view`로 정렬

#### 재시도
- 불필요

### E. 제출 완료 후 미션 참여 화면 진입 시도

#### 발생 상황
- 제출 완료 후 딥링크·수동 경로로 미션 참여 화면 진입 시도

#### 사용자 표시
- 필요 시 짧은 안내 후 오늘의 팀 미션 화면으로 되돌림

#### 상태 처리
- 미션 참여 화면은 표시하지 않음

#### 재시도
- 해당 일·팀에 대해서는 불가

---

## 9-8. 팀 미션 상태 불일치

### A. 내가 제출 완료했는데 화면이 갱신되지 않음

#### 발생 상황
- 제출 성공 후 화면 복귀 실패
- 팀 상태 재조회 실패

#### 사용자 표시
- 토스트

예:
- 제출은 완료되었지만 화면을 새로고침해 주세요

#### 상태 처리
- 오늘의 팀 미션 화면 재조회 버튼 또는 pull-to-refresh
- 재진입 시 최신 상태 로드

### B. 상대 제출 완료 알림이 왔는데 결과 보기 비활성

#### 발생 상황
- 알림은 왔으나 팀 상태 재조회가 늦음
- `result_card` 생성 지연

#### 사용자 표시
- 짧은 안내 문구

예:
- 결과를 준비하는 중이에요. 잠시 후 다시 확인해 주세요

#### 상태 처리
- 버튼 임시 비활성
- 재조회 트리거 허용

#### 재시도
- 가능

---

## 9-9. 결과 보기 실패

### A. 결과 카드 아직 생성 안 됨

#### 발생 상황
- 두 제출은 완료됐으나 서버 후처리 지연
- `result_card open` 지연

#### 사용자 표시
- 전체 페이지 대신 상태 화면

예:
- 결과 카드를 준비하고 있어요
- 잠시 후 다시 시도해 주세요

#### 상태 처리
- 결과 보기 페이지 진입 차단 또는 placeholder 표시

#### 재시도
- 가능

### B. 결과 카드 접근 권한 없음

#### 발생 상황
- `locked` 카드 외부 접근
- 잘못된 링크 접근
- 권한 없는 카드 요청

#### 사용자 표시
- 접근 불가 안내

예:
- 아직 볼 수 없는 카드예요

#### 상태 처리
- 이전 화면 복귀 또는 홈 이동

#### 재시도
- 불필요

### C. 결과 카드 로드 실패

#### 발생 상황
- 네트워크 오류
- API 실패

#### 사용자 표시
- 전체 카드 자리 대신 에러 상태 카드

예:
- 결과를 불러오지 못했어요
- 다시 시도해 주세요

#### 상태 처리
- 다시 시도 버튼 제공

#### 재시도
- 가능

## 9-14. 목록형 empty state 정의

### A. 내 결과 카드 없음

#### 표시 문구
- 아직 참여한 결과 카드가 없어요

#### 액션
- 오늘의 미션 참여하러 가기
- 홈 이동

### B. 저장한 카드 없음

#### 표시 문구
- 아직 저장한 카드가 없어요

#### 액션
- Board 둘러보기
- Board 이동

### C. Board 카드 없음

#### 표시 문구
- 아직 공개된 결과 카드가 없어요

## 9-16. 빌드 / 설정 실패 방지 정책
이 항목은 **런타임 예외보다도 중요하다.**

말씀하신 대로 설정 문제 때문에 앱이 빌드되지 않거나,  
출시 후 특정 기능이 전부 죽는 상황을 막아야 한다.

### 필수 원칙

#### 1. 필수 환경변수 누락 시 앱 실행 금지

대상:
- Supabase URL
- Supabase anon key
- Firebase config
- Sentry DSN(선택적이어도 null-safe 처리)
- app environment

#### 2. 플랫폼별 푸시 설정 누락 검사
- Android FCM key
- iOS APNs / FCM config

#### 3. 라우트 파일 누락 방지
- 모든 네비게이션 도착 지점 화면은 실제 파일 존재해야 함
- 클릭 가능한 메뉴는 placeholder라도 화면 있어야 함

#### 4. 타입 불일치로 빌드 깨지지 않게 enum 고정
- `countryCode`
- `interestKey`
- `reactionType`
- `notificationType`
- `status` 값들

#### 5. null-safe 렌더링 필수
예:
- `profileImageUrl` 없을 때 기본 이미지
- `compareLine` 없을 때 fallback 문구
- `expressionSummary` 비어도 UI 깨지지 않음

## 9-17. 구현 체크리스트
각 기능 구현 시 아래를 반드시 확인해야 한다.

### 체크리스트
- [ ] 실패 시 사용자에게 무엇이 보이는가
- [ ] 실패 후 상태를 롤백해야 하는가
- [ ] 재시도 가능한가
- [ ] 이전 입력값을 유지해야 하는가
- [ ] empty state가 정의되어 있는가
- [ ] 네트워크 실패 시 fallback UI가 있는가
- [ ] 필수 env 누락 시 개발 단계에서 즉시 감지되는가
- [ ] 클릭 가능한 메뉴에 실제 라우트가 존재하는가

## 9-18. 최종 요약
이 앱의 예외 / 실패 처리 원칙은 아래 5줄로 정리된다.

- 실패는 숨기지 말고 짧고 명확하게 알려준다.
- 실패 후 UI와 서버 상태가 어긋나지 않게 롤백 또는 재조회한다.
- 재시도 가능한 실패는 반드시 재시도 경로를 제공한다.
- empty state는 에러가 아니라 정상 상태로 설계한다.
- 환경변수 / 라우트 / 타입 / 푸시 설정 누락은 빌드 / 초기 실행 단계에서 최대한 빨리 감지한다.
