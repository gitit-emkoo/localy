# 권한 / 정책 정의서

## 문서 목적
이 문서의 목적은 아래를 명확히 정의하는 것이다.

- 누가 어떤 데이터를 볼 수 있는지
- 누가 어떤 데이터를 수정할 수 있는지
- 언제 공개되고 언제 비공개인지
- 어떤 행동이 허용되고 어떤 행동이 금지인지
- MVP에서 어디까지 허용하고 어디부터 막을지

이 문서가 없으면 보통 이런 문제가 생긴다.

- 아직 공개되면 안 되는 결과 카드가 Board에 노출됨
- 다른 사람 카드 수정 / 삭제 가능
- 저장 / 반응 중복 허용
- 알림이 엉뚱한 사람에게 감
- 프로필 편집 범위가 과도하거나 부족함

그래서 7번은 **보안 + 제품정책 + UX 규칙**을 같이 묶는 문서라고 보면 된다.

# 7. 권한 / 정책 정의서

## 7-1. 문서 목적
이 문서는 앱의 각 리소스에 대해 아래를 정의한다.

- 조회 권한
- 생성 권한
- 수정 권한
- 삭제 권한
- 공개 시점
- 정책상 제한사항

모든 권한 판단은 MVP 기준으로 정의하며,  
프론트엔드 UI 제한과 백엔드 권한 제한을 둘 다 적용해야 한다.

즉:

- UI에서 숨기는 것만으로 끝내지 말고
- 서버 / DB 레벨에서도 막아야 한다

---

## 7-2. 권한 설계 원칙

### 원칙 1. 본인 데이터는 본인만 수정
예:
- 내 프로필
- 내 관심사
- 내 설정
- 내 제출물
- 내 저장 목록

### 원칙 2. 공개 결과만 전체가 조회 가능
예:
- Board의 카드
- World의 카드 집계

아직 결과가 열리지 않은 데이터는 외부 노출 금지

### 원칙 3. 팀 내부 데이터와 전체 공개 데이터 분리
예:
- 오늘의 팀 화면은 팀 당사자만 접근 가능
- Board 결과 카드는 공개 후 전체 접근 가능

### 원칙 4. MVP에서는 복잡한 공개범위 설정 없음
예:
- 프로필 공개범위 설정 없음
- 카드 공개 ON / OFF 없음
- 기본 정책으로 단순 운영

### 원칙 5. 반응 / 저장 / 알림은 사용자 단위 중복 정책을 가진다
즉:
- 같은 카드를 중복 저장 불가
- 같은 카드에 표현 반응 1개만
- 아이콘 반응도 정책적으로 1개 기준

## 7-3. 리소스별 권한 정의

### A. user_profiles

#### 조회
- 본인: 전체 조회 가능
- 다른 사용자: 결과 카드 및 팀 화면에 필요한 최소 정보만 조회 가능

#### 다른 사용자에게 공개 가능한 필드
- 닉네임
- 국가
- 짧은 프로필
- 프로필 이미지

#### 비공개 필드
- 이메일
- 내부 식별자
- 설정값
- 알림 정보

#### 생성
- 가입 후 자동 생성 가능
- 또는 최초 프로필 설정 완료 시 생성

#### 수정
- 본인만 가능

#### 삭제
- MVP에서는 프로필 개별 삭제 없음
- 계정 탈퇴 시 함께 처리 가능

---

### B. user_interests

#### 조회
- 본인: 전체 조회 가능
- 다른 사용자: 원칙적으로 비공개
- 단, 매칭 설명 용도로 일부 내부 활용 가능

#### 생성
- 본인만 가능

#### 수정
- 본인만 가능

#### 삭제
- 본인만 가능

#### 정책
관심사는 매칭에 쓰이지만, 다른 사용자에게 전면 노출할 필요는 없다.  
MVP에서는 타 사용자 프로필 카드에 관심사 표시하지 않아도 된다.

---

### C. missions

#### 조회
- 로그인 사용자 전체 조회 가능
- `published` 상태의 오늘 미션만 앱에서 사용

#### 생성
- 관리자 / 운영자만 가능

#### 수정
- 관리자만 가능

#### 삭제
- 관리자만 가능

#### 정책
- 일반 유저는 미션 생성 / 수정 / 삭제 불가
- MVP에서는 하루 1개 미션만 `published` 상태

---

### D. match_requests

#### 조회
- 본인 요청만 조회 가능
- 다른 사용자는 조회 불가

#### 생성
- 로그인한 본인만 가능
- 하루 같은 미션 기준 활성 요청 1개만 가능

#### 수정
- 일반 유저 직접 수정 불가
- 상태 전환은 서버 로직(매칭 로직)만 가능

#### 삭제 / 취소
- MVP에서는 취소 기능이 있으면 본인만 가능
- 취소 기능이 없다면 서버 만료 정책으로 종료

#### 정책
- 중복 요청 금지
- 같은 미션에서 `matching` 상태 요청 중복 생성 금지

---

### E. teams

#### 조회
- 팀 당사자 2명만 조회 가능
- 단, 결과 카드가 공개된 이후에는 팀의 공개 필드 일부가 결과 카드로 노출됨

#### 공개 필드
- 국가 조합
- 닉네임
- 짧은 프로필
- 제출 결과

#### 비공개 필드
- 내부 `team id` 자체는 직접 노출 불필요
- 내부 매칭 메타데이터

#### 생성
- 서버 매칭 로직만 가능

#### 수정
- 일반 유저 불가
- 서버만 상태 업데이트 가능

#### 삭제
- MVP에서 일반 삭제 없음

#### 정책
- 팀은 반드시 2명
- 다른 국가끼리만 허용
- 한 미션에 대한 팀은 하루 단위로 유효

---

### F. submissions

#### 조회
- 팀 당사자만 조회 가능
- 결과 카드 공개 전까지 외부 노출 금지

#### 생성
- 팀 당사자 본인만 가능

#### 수정
- MVP에서는 **불가**(제출 후 수정·삭제·재제출 모두 불가)

#### 삭제
- MVP에서는 **불가**

#### 정책
- 팀당 1인 1제출물
- 사진 1장 + 한 줄 설명만
- 제출 후 서버가 팀 상태 갱신

#### 공개 규칙
- 두 제출이 모두 완료되기 전까지 외부 비공개

---

### G. result_cards

#### 조회
- `open` 상태면 전체 로그인 유저 조회 가능
- `locked` 상태면 팀 당사자 외 조회 불가
- Board에는 `open` 상태 카드만 표시

#### 생성
- 서버만 가능
- 두 제출물이 모두 존재할 때 생성 또는 `open` 전환

#### 수정
- 일반 유저 불가
- 서버 또는 운영 도구만 가능

#### 삭제
- MVP에서 일반 삭제 없음

#### 정책
- 결과 카드가 앱의 핵심 공개 객체
- 공개 시점 엄격히 통제
- 내 팀 결과와 타 팀 결과는 같은 `result_card` 구조 사용

---

### H. card_icon_reactions

#### 조회
- 모든 공개 `result_card`에서 집계 조회 가능

#### 생성
- 로그인 유저만 가능

#### 수정
- **다른 아이콘**으로 변경 시 기존 행의 `reaction_type` 갱신(덮어쓰기)

#### 삭제
- **같은 아이콘 재클릭(취소)** 시 해당 행 삭제(MVP 고정)

#### 정책 (MVP 고정)
- 유저당 카드당 1개만 유지
- 같은 아이콘 재탭 = 취소, 다른 아이콘 탭 = 변경
- 중복 카운트 금지

#### 비고
아이콘 반응 자체는 공개 집계 대상이다.  
누가 눌렀는지 개별 목록은 MVP에선 공개 불필요하다.

---

### I. card_expression_reactions

#### 조회
- 공개 카드에서 집계 / 대표 표현 조회 가능

#### 생성

#### 수정
- 표현 변경 허용 가능
- 기존 것 덮어쓰기

#### 삭제
- 표현 취소 기능을 둘 경우 가능
- MVP에선 변경만 허용해도 충분

#### 정책
- 유저당 카드당 1개만 허용
- 자유댓글 금지
- 미리 정의된 문구만 선택 가능

#### 비고
표현 내용은 공개될 수 있으나, 작성자 실명형 노출은 MVP에서 생략 가능하다.

- 본인만 조회 가능

#### 생성
- 본인만 가능

#### 수정
- 없음

#### 삭제
- 본인만 가능

#### 정책

### K. notifications

- 본인만 조회 가능

#### 삭제

#### 정책

- 본인만 조회 가능

#### 생성

#### 수정
- 본인만 가능

#### 삭제
- 일반 삭제 없음

#### 정책
- 알림 on / off
- 언어 설정
- 번역 언어 설정
- 모두 개인 비공개

## 7-4. 화면별 접근 권한

### 홈
- 로그인 사용자만 접근 가능
- 프로필 완료 사용자만 접근 가능

### 오늘의 팀 미션
- 매칭 완료된 본인만 접근 가능
- 다른 유저 접근 불가

### 미션 참여
- 해당 팀의 본인만 접근 가능
- 매칭 없는 사용자는 접근 불가
- **당일 해당 미션·팀에 대해 제출이 완료된 뒤에는 미션 참여(입력) 화면으로 재진입 불가**(오늘의 팀 미션 화면만 사용)

### 결과 보기

#### 내 팀 결과
- 팀 당사자 접근 가능
- 공개 전에도 접근 가능 여부는 정책 필요

추천:
- 둘 다 완료되기 전에는 접근 불가
- `ready_to_view` 이후 접근 가능

#### 타 팀 결과
- 공개 카드만 접근 가능

### Board
- 공개 카드만 노출
- 로그인 사용자만 접근 가능

### World
- 공개 집계 정보만 노출
- 로그인 사용자만 접근 가능

### My
- 본인만 접근 가능

### 내 결과 카드 목록
- 본인만 접근 가능

### 저장한 카드 목록
- 본인만 접근 가능

### 알림
- 본인만 접근 가능

### 프로필 편집
- 본인만 접근 가능

### 문의하기
- 로그인 사용자만 접근 가능

## 7-5. 프로필 공개 정책
MVP에서는 너무 복잡한 공개범위를 두지 않는다.

### 공개되는 프로필 요소
- 닉네임
- 국가
- 짧은 프로필
- 프로필 이미지

### 비공개 유지
- 이메일
- 설정
- 관심사 상세
- 알림 정보
- 내부 식별값

### 이유
팀 경험과 결과 카드 소비에 필요한 최소 정보만 공개하고,  
개인 식별성과 프라이버시는 최대한 제한한다.

---

## 7-6. 결과 카드 공개 정책
이 부분은 매우 중요하다.

### 공개 조건
- 팀의 두 제출물이 모두 완료
- `result_card.status = open`

### 공개 후
- Board 노출 가능
- World 집계 반영 가능
- 다른 사용자 반응 가능
- 저장 가능

### 비공개 조건
- 한쪽이라도 미제출
- `result_card.status = locked`

### 비공개 상태에서 금지
- Board 노출 금지
- World 집계 반영 금지
- 타 사용자 접근 금지

---

## 7-7. 반응 정책

### 아이콘 반응
- 로그인 사용자만 가능
- 공개 결과 카드에서만 가능
- 유저당 카드당 1개만 유지(MVP: 같은 아이콘 재탭 = 취소, 다른 아이콘 = 변경)

### 표현 반응
- 로그인 사용자만 가능
- 공개 결과 카드에서만 가능
- 유저당 카드당 1개만 허용
- 자유댓글 금지
- 선택형 문구만 허용

### 저장
- 공개 결과 카드에서만 가능
- 저장 목록은 개인 비공개

---

## 7-8. 신고 / 문의 / 차단 정책
MVP에서는 최소 정책만 둔다.

### 문의하기
- 로그인 사용자만 가능
- 텍스트 제출

### 신고하기
- 결과 카드 또는 사용자에 대해 추후 확장 가능
- MVP에서 UI가 없으면 제외 가능

### 차단
- MVP에서 반드시 필요하지는 않음
- 정식버전에서 확장 가능

즉, MVP에선:
- 문의하기 필수
- 신고 / 차단은 후순위 가능

---

## 7-9. 수정 가능 범위 정책

### 사용자가 수정 가능한 것
- 프로필
- 관심사
- 설정
- 저장 목록
- 알림 read 상태

### 사용자가 수정 불가능한 것
- 미션
- 팀 구성
- 결과 카드 본체
- 다른 사람 제출물
- 다른 사람 반응
- 알림 원문 내용

---

## 7-10. 삭제 정책
MVP에서는 삭제 정책을 최소화한다.

### 허용
- 저장 해제
- 선택형 표현 반응 변경(MVP: 표현 1개 유지·변경)
- 아이콘 반응 취소(MVP: 같은 아이콘 재탭)

### 비허용 추천
- 제출물 삭제
- 결과 카드 삭제
- 팀 삭제
- 미션 삭제

### 이유
MVP에서 삭제 허용 범위를 넓히면 상태 복잡도가 크게 증가한다.

---

## 7-11. 매칭 정책

### 필수 조건
- 서로 다른 국가
- 같은 날짜 / 같은 미션
- 조건형 랜덤 매칭
- 시간 민감형은 시차 고려
- 동일 사용자 중복 매칭 금지

### 유저 권한
- 유저는 매칭 요청만 생성 가능
- 실제 매칭 확정은 서버만 가능

## 7-12. 알림 권한 정책

### 유저가 제어 가능한 것
- 푸시 on / off
- 알림 타입별 on / off

### 유저가 제어 불가능한 것
- 알림 생성 자체
- 타인 알림 열람
- 알림 내용 수정

---

## 7-13. 구현 시 강제 규칙
이건 매우 중요하다.

### 규칙 1
프론트에서 숨겼다고 권한 처리 끝난 것으로 간주하지 말 것  
반드시 백엔드 / DB 정책까지 설정

### 규칙 2
공개 / 비공개 판정은 반드시 `result_cards.status` 기준으로 할 것

### 규칙 3
유저 소유 리소스는 `user_id` 기준으로만 접근 허용

### 규칙 4
매칭 / 팀 / 결과카드 생성은 서버만 수행  
클라이언트에서 임의 생성 금지

### 규칙 5
저장 / 반응은 unique 제약으로 중복 방지할 것

---

## 7-14. Supabase 기준 권한 구현 방향
Supabase를 사용할 경우, Row Level Security(RLS)를 통해 본인 데이터와 공개 데이터의 접근을 제어하는 구조가 적합하다.

즉 구현 방향은 다음이 적합하다.

### 본인만 접근 가능
- `user_profiles`
- `user_settings`
- `saved_cards`
- `notifications`

### result_cards
- `open` 상태만 전체 조회 가능
- `locked`는 팀 당사자만

### submissions
- 본인 및 팀 맥락에서만 접근
- 외부 공개 금지

### card_icon_reactions / card_expression_reactions
- 공개 카드에 대해서만 생성 / 조회

---

## 7-15. 최종 요약
이 앱의 권한 / 정책 구조는 아래 5줄로 요약된다.

- 프로필 / 설정 / 저장 / 알림은 개인 자산이므로 본인만 접근
- 팀 정보와 제출물은 팀 당사자 중심으로 제한
- 결과 카드는 둘 다 제출 완료 후에만 공개
- 반응과 저장은 공개 카드에서만 가능하고 중복은 금지
- 공개 / 비공개와 수정 가능 여부는 프론트가 아니라 서버 / DB 정책까지 포함해 강제
