# 상태 정의서 / 구현 안정성 규칙서

## 문서 개요
특히 이 앱은 화면보다도 상태 전이가 핵심이다.

핵심 상태 축:
- 매칭 전 / 매칭 중 / 매칭 완료
- 미제출 / 내 제출 완료 / 둘 다 완료
- 결과 잠금 / 결과 공개

핵심 관리 항목:
- 푸시 발송 조건
- 버튼 활성 / 비활성 조건

빌드 안정성 측면에서는 Expo가 파일 기반 라우팅을 제공하는 Expo Router를 지원하고, TypeScript를 1급으로 지원하며, EAS Build와 환경변수 체계를 공식적으로 제공하므로,  
이 앱은 **Expo + TypeScript + Expo Router + EAS Build** 기준으로 문서를 작성하는 것이 가장 안전하다.

Supabase는 Postgres / Auth / Storage / Realtime / Edge Functions를 한 플랫폼에서 제공하고,  
FCM은 크로스플랫폼 푸시를 제공하므로, 이 조합이 현재 MVP 구조와 가장 잘 맞는다.

아래 문서는 기획자의 의도를 정확히 이해하고, 설정 문제로 빌드가 깨지지 않게 하기 위한 **5번 상태 정의서 + 구현 안정성 규칙**이다.

---

## 5-1. 문서 목적
이 문서는 글로벌 미션 앱의 모든 핵심 상태를 정의하고,  
각 상태에서 UI가 어떻게 보이고 어떤 동작이 가능한지, 어떤 서버 이벤트가 발생하는지를 고정한다.

이 문서의 목적은 다음 4가지다.

- 화면과 서버 상태를 같은 용어로 통일
- 버튼 / 모달 / 토스트 / 알림 조건을 명확히 정의
- 절대 임의 해석으로 상태를 섞지 못하게 차단
- 빌드 / 배포 시 설정 누락, 환경변수 누락, 라우팅 오류를 줄임

---

## 5-2. 구현 원칙
이 앱은 화면 중심 구현이 아니라 **상태 중심 구현**으로 만든다.

즉 다음 순서로 생각해야 한다.

1. 현재 상태가 무엇인가
2. 이 상태에서 보여야 하는 UI는 무엇인가
3. 이 상태에서 가능한 액션은 무엇인가
4. 액션 이후 어떤 새 상태로 전이되는가

### 절대 금지할 것들
- 화면별로 개별 로직 중복 작성
- 같은 상태를 화면마다 다른 이름으로 표현
- 버튼 클릭 시 서버 상태 업데이트 없이 UI만 바꾸는 방식
- 상태 정의 없이 임의 조건문으로만 흐름 구현

## 5-3. 전체 상태 도메인
이 앱은 상태를 아래 7개 도메인으로 나눈다.

- 인증 상태
- 온보딩 / 초기설정 상태
- 매칭 상태
- 팀 미션 상태
- 결과 카드 상태
- 반응 / 저장 상태
- 알림 상태

각 도메인은 서로 영향을 주지만, 이름과 책임은 분리한다.

---

## 5-4. 인증 상태 정의

### 상태값
- `signed_out`
- `signed_in`

### 의미

#### `signed_out`
- 로그인 / 가입 전 상태
- 이메일 입력 화면만 진입 가능

#### `signed_in`
- 인증 완료 상태
- 이후 초기 프로필 설정 여부를 추가 판단해야 함

### UI 규칙
- `signed_out`이면 홈 / Board / World / My 진입 불가
- `signed_in`이면 초기설정 여부에 따라 다음 화면 결정

---

## 5-5. 초기 설정 상태 정의

### 온보딩(MVP) 정의
- 별도 소개 슬라이드형 온보딩 화면은 없음
- 이메일 가입·로그인과 최초 프로필·관심사 설정 흐름을 **온보딩**이라 통칭할 수 있음

### 상태값
- `profile_incomplete`
- `profile_completed`

### 의미

#### `profile_incomplete`
- 최초 가입 직후
- 닉네임, 나이, 국가, 한 줄 프로필, 관심사 최소 1개가 아직 미완성

#### `profile_completed`
- 필수 프로필 완료
- 앱 사용 가능

### 이동 규칙
- `signed_in + profile_incomplete` → 반드시 프로필 / 관심사 설정 화면
- `signed_in + profile_completed` → 홈 진입

### UI 규칙
- `profile_incomplete` 상태에서는 하단 탭 접근 금지
- 이 상태에서는 시작하기 버튼이 필수 조건 충족 전까지 비활성

## 5-6. 매칭 상태 정의

### 상태값
- `idle`
- `matching`
- `matched`
- `match_failed`

### 의미

#### `idle`
- 오늘 미션은 열려 있으나 아직 매칭 요청 전

#### `matching`
- 유저가 매칭 시작을 클릭했고, 상대 국가 유저를 찾는 중

#### `matched`
- 오늘의 팀 생성 완료
- 상대 유저가 확정됨

#### `match_failed`
- 일정 시간 안에 매칭이 실패했거나 서버 오류 발생

### 홈 오늘의 미션 카드와의 연결 규칙

#### 상태 `idle`
표시:
- 버튼명: `매칭 시작`
- 버튼: 활성
- 상태 문구: `오늘의 팀을 아직 찾지 않았어요`

클릭:
- 매칭 시작 확인 모달 오픈

#### 상태 `matching`
표시:
- 버튼명: `매칭 중...`
- 버튼: 비활성
- 상태 문구: `비슷한 시간대의 다른 국가 팀원을 찾고 있어요`

추가 규칙:
- 앱 종료 가능
- 서버에서 매칭 완료되면 푸시 발송

#### 상태 `matched`
표시:
- 버튼명: `오늘의 미션 참여`
- 버튼: 활성
- 상대 국가 / 국기 노출
- 상태 문구: `오늘의 글로벌 팀이 준비되었어요`

클릭:
- 오늘의 팀 미션 화면 이동

#### 상태 `match_failed`
표시:
- 버튼명: `다시 매칭하기`
- 버튼: 활성
- 상태 문구: `팀을 찾지 못했어요. 다시 시도해 주세요`

클릭:
- 매칭 재시도
- 또는 매칭 실패 모달

### 서버 규칙
- 매칭 요청은 유저당 하루 1개 활성 요청만 허용
- `matching` 중복 요청 금지
- `matched`이면 새 매칭 시작 금지
- 매칭 생성은 Edge Function에서 처리
- 클라이언트는 직접 상대를 매칭하지 않음

## 5-7. 팀 미션 상태 정의

### 상태값
- `not_started`
- `self_submitted_waiting`
- `peer_submitted_waiting`
- `ready_to_view`

### 의미

#### `not_started`
- 나도 상대도 제출 전
- 또는 둘 다 제출 전 상태를 대표

#### `self_submitted_waiting`
- 나는 제출 완료
- 상대는 아직 미제출

#### `peer_submitted_waiting`
- 상대는 제출 완료
- 나는 아직 미제출

#### `ready_to_view`
- 둘 다 제출 완료
- 결과 카드 공개 가능

### 오늘의 팀 미션 화면 규칙

#### 상태 `not_started`
표시:
- 진행률: `0/2`
- 나: 아직 업로드 전
- 상대: 아직 업로드 전
- 버튼명: `미션하기`
- 버튼: 활성

클릭:
- 미션 참여 화면 이동

#### 상태 `self_submitted_waiting`
표시:
- 진행률: `1/2`
- 나: 업로드 완료
- 상대: 아직 업로드 전
- 버튼명: `결과 보기`
- 버튼: 비활성
- 보조 문구: `상대 팀원의 업로드가 완료되면 결과 보기가 활성화됩니다`

#### 상태 `peer_submitted_waiting`
표시:
- 진행률: `1/2`
- 나: 아직 업로드 전
- 상대: 업로드 완료
- 버튼명: `미션하기`
- 버튼: 활성
- 보조 문구: `상대 팀원은 이미 답변을 완료했어요`

클릭:
- 미션 참여 화면 이동

#### 상태 `ready_to_view`
표시:
- 진행률: `2/2`
- 나: 업로드 완료
- 상대: 업로드 완료
- 버튼명: `결과 보기`
- 버튼: 활성

클릭:
- 결과 보기 화면 이동

## 5-8. 미션 참여 상태 정의

### 상태값
- `draft_empty`
- `draft_partial`
- `ready_to_submit`
- `submitting`
- `submit_success`
- `submit_failed`

### 의미

#### `draft_empty`
- 사진 없음
- 설명 없음

#### `draft_partial`
- 사진만 있거나 설명만 있음

#### `ready_to_submit`
- 사진 1장 + 설명 1줄이 모두 존재

#### `submitting`
- 제출 API 호출 중

#### `submit_success`
- 제출 성공

#### `submit_failed`
- 제출 실패

### 미션 참여 화면 UI 규칙

#### `draft_empty`
- 제출 버튼 비활성

#### `draft_partial`
- 제출 버튼 비활성

#### `ready_to_submit`
- 제출 버튼 활성

#### `submitting`
- 제출 버튼 로딩
- 중복 클릭 금지

#### `submit_success`
- 토스트: `제출이 완료되었어요`
- 오늘의 팀 미션 화면으로 복귀
- 팀 미션 상태 갱신
- 당일 해당 미션·팀에 대해 **미션 참여(입력) 화면 재진입 불가**(라우팅에서 차단)

#### `submit_failed`
- 실패 모달 또는 에러 토스트
- 재시도 가능

### 이탈 규칙
사진 또는 설명이 하나라도 입력된 상태에서 뒤로가기:
- 작성 중 이탈 확인 모달 오픈

## 5-9. 결과 카드 상태 정의

### 상태값
- `locked`
- `open`

### 의미

#### `locked`
- 둘 다 제출 완료 전
- 결과 카드 비공개

#### `open`
- 둘 다 제출 완료
- 결과 카드 공개 가능

### 규칙
- 결과 카드 레코드는 `open`일 때만 Board에 노출
- `locked` 상태 결과 카드는 외부 목록에 절대 노출하지 않음
- 내 팀 화면에서는 `locked`일 때 결과 보기 버튼 비활성

## 5-10. 반응 상태 정의

### 1) 공통 아이콘 반응

#### 규칙 (MVP 고정)
유저당 카드당 **동시에 하나의 아이콘 반응만** 존재한다(행 1개 또는 동일 `(result_card_id, user_id)` 유니크).

#### MVP 정책 (확정)
- **같은 아이콘을 다시 누르면** 반응이 **취소**된다(집계에서 제거).
- **다른 아이콘을 누르면** 기존 반응은 **새 아이콘 반응으로 변경**된다(덮어쓰기).

이렇게 해야 중복 카운트 오류를 막기 쉽다.

### 2) 표현 반응

#### 상태값
- `not_expressed`
- `expressed`

#### 규칙
- 유저당 카드당 표현 반응 1개만 허용
- 표현 변경은 가능하지만, 중복 누적 금지
- 표현은 자유댓글이 아니라 선택형 문구만 허용

#### UI
- 표현 선택 버튼 → 바텀시트 오픈
- 선택 완료 → 토스트
- 표현 수 즉시 갱신

### 3) 저장 상태

#### 상태값
- `unsaved`
- `saved`

#### 규칙
- 유저당 카드당 저장 1개
- 저장 토글 가능

#### UI
- 저장 시 토스트
- 저장 해제 시 토스트

## 5-11. 알림 상태 정의

### 알림 이벤트 고정
- 매칭 완료
- 상대 제출 완료
- 새 반응 도착(묶음형)
- 오늘의 새 미션 오픈(후순위)

### 알림 타입 상태
- `unread`
- `read`

### 이동 규칙

#### 매칭 완료 알림 클릭
- 오늘의 팀 미션 화면 이동

#### 상대 제출 완료 알림 클릭
- 결과 보기 화면 이동

#### 새 반응 도착 알림 클릭
- 해당 카드 결과 보기 화면 이동

#### 오늘의 새 미션 알림 클릭
- 홈 이동

### 푸시 / 인앱 동시 규칙
같은 이벤트는 아래처럼 동작해야 한다.

- 앱 밖: 푸시
- 앱 안: 알림 목록 기록

## 5-12. 빈 상태 정의

### 꼭 필요한 empty state
- 내 결과 카드 없음
- 저장한 카드 없음
- Board 카드 없음
- World 데이터 없음
- 알림 없음

### 규칙
각 empty state는 반드시 아래를 포함해야 한다.

- 짧은 설명
- 다음 행동 버튼

예:
- `아직 저장한 카드가 없어요`
- `Board 둘러보기`

### 필수 에러 상황
- 매칭 실패
- 제출 실패
- 반응 실패

## 5-14. 라우팅 상태 규칙

### 라우팅 원칙
- 탭: 홈 / Board / World / My
- 스택: 프로필 편집, 결과 보기, 미션 참여, 문의하기, 알림 설정 등
- 결과 보기 화면은 단일 라우트 1개만 사용
- 내 팀 결과와 타 팀 결과를 같은 화면에서 처리

### 금지
- 내 결과용 상세와 타 팀용 상세를 별도 파일로 분리
- 같은 개념의 화면을 두 개로 복제

## 5-15. 환경변수 / 빌드 안정성 규칙
빌드 실패를 막기 위해 이 부분은 반드시 명문화해야 한다.

### 원칙

#### 1. 환경변수는 하드코딩 금지
예:
- Supabase URL
- Supabase anon key
- Firebase config
- Sentry DSN
- 앱 환경 구분값(dev / staging / prod)

#### 2. app config와 build profile 분리
- `eas.json`으로 dev / preview / production 프로필 분리
- 환경별 앱 이름, 번들 ID, API endpoint 분리

#### 3. 필수 env 누락 시 앱 실행 차단
앱 시작 시 필수 환경변수가 없으면:
- 조용히 진행 금지
- 개발 환경에서는 명확한 에러 throw

#### 4. 플랫폼별 설정 분리
- iOS / Android 푸시 설정 별도 관리
- FCM / APNs 키 누락 시 빌드 전 체크

#### 5. New Architecture 관련 규칙
- MVP에서는 외부 네이티브 의존성을 최소화
- 필요 시 호환성 검증 후 활성화

## 5-17. 최종 요약
이 앱 구현에서 가장 중요한 것은 화면이 아니라 **상태 전이의 일관성**이다.

한 줄로 압축하면:

- 로그인하면 초기설정 여부를 먼저 판단
- 홈에서는 매칭 상태가 오늘의 미션 버튼 상태를 결정
- 팀 미션 화면에서는 제출 상태가 메인 버튼 상태를 결정
- 둘 다 제출 완료되어야 결과 카드가 공개
- 결과 카드는 하나의 상세 화면으로 통합
- 반응 / 저장 / 알림은 모두 중복 방지 규칙을 가진다
- 환경변수와 빌드 프로필은 처음부터 분리해 설정 누락으로 인한 빌드 실패를 막는다
